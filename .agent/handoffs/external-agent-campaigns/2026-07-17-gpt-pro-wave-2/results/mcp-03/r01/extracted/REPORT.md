## Decisions that define the target

The default read profile is exactly six transactions:

| Role | Discovered transaction tools | Count |
|---|---|---:|
| `read` | `query`, `read`, `get`, `explain`, `context`, `status` | 6 |
| `write` | read set + `write`, `run` | 8 |
| `review` | write set + `judge` | 9 |
| `admin` | review set + `maintenance` | 10 |

The landed registry currently declares seven read transactions because `graph` is separate even though its semantic verb is already `get`. The six-tool design folds it into `get(ref, view="graph.*", direction, page)`. This preserves recursive continuation while removing a competing discovery choice.

Canonical request shapes:

```text
query(plan, projection, semantics, page)
read(ref, view, page)
get(ref, view="default", direction=null, page=null)
explain(subject, aspect, input=null)
context(seed, policy, budget, recipient_ref=null, successor_session_id=null, page=null)
status(domain, ref=null, detail=null, page=null)

write(operation, ref, body, authorization, idempotency_key=null, dry_run=false)
judge(candidate_refs, decision, authorization, idempotency_key=null)
run(recipe_ref, parameters, authorization, page=null, idempotency_key=null)
maintenance(action, ref, body, authorization, dry_run=true, idempotency_key=null)
```

Every response uses one canonical envelope. `items` is the only top-level collection slot; single objects and aggregates are under `data`; `page` states limit/cursor/next cursor/total/exactness/semantics; `result_ref` pins a resumable logical result; `meta` carries archive epoch, coverage, degradation, and schema version; governed operations add a durable `receipt`. Old field names remain inside typed domain objects only when they are domain vocabulary, not competing envelope grammar.

There are **no runtime aliases**. Old names exist only in the equivalence declaration, frozen old goldens, telemetry normalization, and historical documentation. A client that calls an old tool after cutover receives MCP unknown-tool discovery failure, which is intentional under the hard-rename policy.

## Client migration guide

### Repository-owned consumers observed in the snapshot

`polylogue/mcp/server_prompts.py` must stop emitting old call sequences. Each workflow prompt should emit the six-tool expressions in the table. The current `unacknowledged_failures` prompt is especially unsafe as discovery guidance: it is declared/read-visible but tells clients to call `list_marks` and `list_annotations`, both write-role tools. Preserve authority by raising that prompt's minimum role to `write`, or redesign it around a read-authorized acknowledgment projection; do not silently expose personal-state reads to `read`.

`docs/mcp-reference.md` and `docs/mcp-integration.md` must replace the ~100-tool/sample tables with the six read transactions, four privileged transactions, canonical URI resources, explicit result semantics, cursor examples, and role profiles. `docs/hooks.md` and `docs/glossary.md` must replace `compose_context_preamble`/`compile_context` with `context(policy="session-start"|...)` while preserving `successor_session_id`. `CLAUDE.md` currently says 103 tools and describes the old update ritual; rewrite it around declaration generation, the independent six-read/ten-admin role baselines, and parity capture.

`tests/infra/mcp.py` needs exact independent baselines for tools **and** resources/prompts. Today resources/prompts are lower-bound checks: `test_server_surface_contract` rejects extra tools but does not reject extra resources or prompts. That is insufficient for a hard cutover. Add exact role-scoped inventories, including canonical object resources and all retained prompts.

Historical analytics fixtures such as `mcp__polylogue__search` in insight/devtools tests must generally remain as old-log compatibility evidence. Update runtime allowlists and examples, but preserve parsing/grouping of historical qualified names and map them to canonical transaction families in reports.

## Acceptance boundary
