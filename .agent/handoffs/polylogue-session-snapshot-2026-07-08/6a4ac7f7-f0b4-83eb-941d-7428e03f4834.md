Projection is less general: `ProjectionSpec` has families, fields, body policy, limits/windows, roles, block exclusions, redaction, etc., but the product still leans on named views like `summary`, `transcript`, `dialogue`, `raw`, `context`, `chronicle`, `neighbors`, and so on. Render is comparatively clean: format, destination, layout, timestamp policy, out path.

The CLI is still a thick surface. The transcript already captured the key B7 finding: `cli/archive_query.py` is the center of gravity, around 2K+ lines, with a parallel query/read engine rather than a purely thin adapter over the facade/protocol. It opens archive stores, shapes results, and mirrors behavior that should belong to the read contract. That makes Go/Rust/client-language work premature until `t46` has actually thinned the CLI. fileciteturn1file4

The daemon already exposes HTTP routes, query completions, action affordances, status, web reader routes, import scheduling, maintenance routes, and a browser-capture surface. It is not yet the mandatory substrate for all read/write UX. It uses a TCP HTTP API and pidfile/advisory-lock lifecycle. There is no first-class UDS protocol, no composer protocol, no cursor-aware completion protocol, and no live preview protocol. The current `/api/query-completions` endpoint is registry metadata, not “complete this partial expression at cursor X against this archive state.”

The CLI’s “bare query” semantics have already moved in the direction you wanted: the code now has a strict floor for unquoted unsignalled bare roots. Explicit `find`, `--`, quoted expressions, or structurally clear expressions can query; an unquoted lone typo should be rejected before an expensive search. That means the typo-hint / empty-search path is no longer the important UX story. The important story is structural composition, dynamic completion, and preview.

## Report A2 — daemon local protocol

The local protocol should be designed as the spine of the whole experience, not as a small convenience endpoint. The daemon is the warm process that owns parsing, archive identity, completion, preview, query execution, render/projection expansion, action affordances, and degraded-state truth. Thin clients should never reimplement query parsing, SQL routing, archive filtering, projection semantics, or readiness logic.

The protocol should be versioned, local-first, archive-bound, and cancellation-aware. The initial transport can be UDS HTTP to reuse current daemon machinery, but the envelope should be RPC-shaped from day one: request id, method, params, client capabilities, archive identity, deadline, optional cancellation token, and response metadata. JSON-RPC is a reasonable conceptual model because it already standardizes request/response/notification shapes and is transport-agnostic; LSP shows how far that model can go for completion/diagnostics/preview-like workflows. citeturn844412search2turn844412search18

Recommended methods:

`daemon.hello` returns daemon version, archive root fingerprint, schema versions, capabilities, protocol version, started_at, degraded reasons, and convergence epochs.

`query.parse` returns AST, relation grain, projection clause, pipeline stages, diagnostics, and expected-next-token sets for completion.

`query.complete` accepts full input plus cursor byte offset, not just `kind=incomplete`. It returns replacement span, ranked candidates, candidate kind, source, count/sample when cheap, and explanation.

`query.preview` accepts a full or partial query/projection/render spec and returns a bounded preview envelope: grain, rows, count/exactness, degraded state, cost estimate, cache key, freshness epoch, and timing.

`query.execute` executes the committed query. It is separate from preview because preview can use cheaper approximations and smaller limits.

`projection.preview` renders a small representative projection, including “what would be omitted/folded/redacted.”

`value.complete` returns archive-backed values for a field, with counts and freshness.

`action.affordances` returns legal actions for the current result grain and selected row set.

`daemon.status`, `archive.identity`, `convergence.status`, and `events.subscribe` support live UI feedback.

Errors must be typed and product-facing. A query parse error is not an HTTP 500. A stale index is not “no results.” A partial preview is not a successful full answer. Every response should carry a `degraded` block when relevant, matching the broader `cpf.4` degrade-loudly doctrine.

The protocol must carry archive identity. A thin client must refuse or warn when it is connected to a daemon serving a different archive root/schema/branch than the client expected. That safety guard matters more if the daemon becomes mandatory.

Security posture: UDS under `$XDG_RUNTIME_DIR/polylogue/<archive-hash>/daemon.sock`, 0600 socket permissions, per-user daemon, loopback/TCP kept for browser or remote surfaces only, and explicit token/auth for anything non-UDS. The existing daemon threat-model work should be extended rather than bypassed.

## Report A3 — daemonless inventory

Daemonless support is mixed today. Reads can work without the daemon because the Python API / archive-direct helpers can open the archive directly. Import does not really work daemonless: the import command stages into the daemon inbox and requires the running daemon to accept scheduling. That matches your intuition: “daemonless” is not a full product contract today.

The recommendation is to stop treating daemonless as first-class UX. Keep it as a break-glass and CI mode, not as the normal product path. The product path should be: daemon required, client thin, daemon autostarted/ensured if absent, direct archive path used only for tests, migrations, emergency reads, and maybe `polylogue doctor --offline`.

This simplifies almost everything. Import does not need two truths. Query/read does not need direct-vs-daemon parity at every UI point. Completion and preview can assume a warm parser and warm archive. The CLI can be a client instead of an archive app. The downside is onboarding: a user must understand “Polylogue includes a daemon.” That is solvable with lifecycle design and a good `doctor`/`ensure` command.

The right doctrine is: “Polylogue is daemon-backed by default; daemonless is diagnostic.” Not “daemonless must remain equivalent.”

## Report A4 — mandatory-daemon lifecycle

Mandatory daemon does not mean brittle daemon. The lifecycle has to be boring, self-healing, and archive-scoped.

The identity unit should be one daemon per archive root, not one global daemon for everything. Multiple archives can coexist; their sockets/pidfiles are derived from archive-root fingerprints. The client handshake validates that it is talking to the right archive. If not, it either finds the correct socket or starts the correct daemon.

Startup path:

`polylogue` client starts and does a cheap UDS probe. If no socket exists, it calls `polylogue daemon ensure` / `polylogued ensure` or equivalent. Ensure validates archive root, creates runtime dir, starts daemon detached, waits for `daemon.hello`, then returns control to the client. For scripts, a `--no-autostart` flag can make absence a hard error.

States should be explicit: `starting`, `schema-blocked`, `degraded-running`, `converging`, `ready`, `draining`, `stopping`. This is essential for composer UX: the user can still compose queries while the archive is converging, but preview responses must say “index incomplete” or “semantic lane not ready.”

Crash recovery: stale socket/pidfile cleanup, advisory lock as current guard, ops event on restart, convergence rescan, and no assumption that a previous convergence stage completed. This lines up with the existing automagic-invariants work.

Shutdown should be graceful but bounded. If the daemon owns long-running preview/convergence requests, clients need cancellation and drain semantics. The daemon must not block forever because a TUI client vanished mid-preview.

Prior art strongly supports this shape. Watchman’s CLI attempts to talk to the persistent per-user server and starts it if it does not exist; gopls supports a shared daemon model where forwarder processes can start or forward to the resident daemon. citeturn844412search1turn844412search0

## Report A5 — prior art fast daemons

The useful pattern is not “make a CLI faster.” It is “split client and resident authority.”

Stealable patterns:

gopls: one shared daemon, thin forwarder processes, protocol ownership in the server. This is almost exactly the Polylogue target: editor/CLI clients are cheap; the daemon is warm and semantically authoritative. citeturn844412search0

Watchman: the executable contains both client and server components, but the default user experience is “talk to existing server; start if missing.” That is the lifecycle pattern Polylogue wants. citeturn844412search1

LSP: request/response/notification with cancellation, partial results, diagnostics, and server-side knowledge. Polylogue’s composer is query-LSP-like: completions, diagnostics, preview, explain, and execute over a structured language. citeturn844412search2

Textual / prompt_toolkit / Bubble Tea / Ratatui: these are candidate front-end frameworks, but they are not the architecture. Textual fits fast Python iteration and Polylogue already has a Textual TUI. `prompt_toolkit` is strong for command-line editing/completion. Bubble Tea gives Go a mature Elm-style TUI loop. Ratatui gives Rust a lightweight rich terminal UI. citeturn844412search3turn319747search0turn319747search1turn319747search2

The conclusion: daemon protocol first, front-end framework second. A front-end chosen before the protocol will either reimplement substrate logic or become throwaway.

## Report B7 — CLI thinning worklist

The CLI thinning program should be treated as `t46`’s first real spine, and it gates everything else.

The main pathology is not just imports or latency. It is semantic duplication. `cli/archive_query.py` is effectively another read engine. Daemon HTTP also has split-archive fast paths that mirror query filters instead of routing through one `SessionQuerySpec`/facade path. This is why surfaces diverge. The goal is not “make CLI prettier.” The goal is to delete the parallel engines.

Worklist:

First, classify every CLI direct reach into storage/pipeline/archive. The buckets are: already expressible through `Polylogue`; needs a facade method; presentation-only; diagnostic/break-glass exception. Anything in the first two buckets must leave the CLI.

Second, stabilize the contract: `list/search/facets/read/messages/query_units/query_completions/explain_query/render_projection/action_affordances/import.schedule/maintenance.preview/maintenance.execute/reset.preview/reset.execute`.

Third, convert read paths. `find`, `read`, `facets`, `messages`, query-unit rows, group/count, and selection actions should all accept/produce shared envelopes. The CLI can render envelopes; it should not own retrieval semantics.

Fourth, convert write/maintenance paths. Import already goes through the daemon conceptually; make it protocol-native. Reset and maintenance need typed dry-run/execute contracts.

Fifth, delete compatibility shims and prove parity. Golden tests should compare old/current outputs for demo corpus where compatibility is intentionally preserved, and grep tests should prove direct storage imports are gone from CLI modules except allowlisted diagnostics.

The most important implementation principle: never add a new thin path while keeping the old thick path as production default. That is the current dual-path trap. Replacement must be same-PR deletion.

## Report B8 — client/substrate contract

The client/substrate contract should be expressed as a set of typed protocol models, not “whatever the daemon HTTP happens to return.”

Core models:

`ArchiveIdentity`: archive_root hash, tier schema versions, daemon version, branch/commit when available, configured sources, convergence epochs.

`QueryInput`: text, cursor offset, mode, default grain, selected macro/view context, root filters, limit, retrieval lane.

`QueryParseEnvelope`: AST payload, relation grain, stage list, projection clause, diagnostics, expected completions, normalized string, explain text.

`CompletionRequest` / `CompletionEnvelope`: input text, cursor, scope, prefix, candidate list with replace span, kind, source, score, count, stale/degraded flags.

`PreviewRequest` / `PreviewEnvelope`: spec, requested budget, rows or rendered fragment, exactness, timing, cache key, freshness, degraded state.

`ExecutionEnvelope`: committed output, stable machine envelope, route/provenance, warnings.

`ActionRequest` / `ActionDryRun` / `ActionResult`: result-set identity, action kind, safety guard, dry-run diff, confirmation token, execution result.

`ProjectionProgram` and `RenderProgram`: not just string flags. The client can construct these interactively and the daemon validates/executes them.

Thin clients should be allowed to do local presentation logic and shallow keystroke buffering. They should not perform archive reads, SQL filtering, DSL lowering, projection expansion, value enumeration, or mutation planning. If a client must know something semantic, that knowledge should come from a daemon capability/registry endpoint.

This contract is also the bridge to Go/Rust. A Go/Rust client can speak this protocol. It cannot sensibly “use the Python API” without reintroducing Python startup and deployment complexity.

## Report C10 — composer UX design

The composer should not be a fancier shell completion. It should be a structural workbench for composing a relation, projection, render, and optional action.

Visual model:

Top line: editable query expression.

Left panel: structural completion candidates, grouped by what the grammar expects at the cursor: fields, operators, values, units, pipeline stages, projection units, render layouts, actions.

Right/top preview: live result sample, count, exactness, degraded status.

Right/bottom or secondary pane: explain/plan, showing parsed AST, relation grain, retrieval lane, pipeline stages, projection program, and cost/readiness warnings.

Footer: legal next actions: read, group, count, save macro, save cohort, export, copy command.

The composer has modes, but modes are compositional axes, not separate screens:

Selection mode answers “which rows?”

Projection mode answers “which evidence from those rows?”

Render mode answers “how should this be displayed/exported?”

Action mode answers “what can I do with this result set?”

The composer should always preserve the text form. The user is not trapped in a form builder. Every UI decision edits a command/query string or a serialized spec. This is crucial: the CLI remains scriptable, the composer teaches the DSL, and macros/views can be saved from exactly what the user built.

Good first vertical slice: session-grain selection only. The first version does not need set algebra, projection algebra, or saved cohorts. It needs cursor-aware completion, value completion, and preview for `find QUERY` + `| group by` + `| count` + `| read`. Then expand.

## Report C11 — progressive preview semantics

Preview is not “execute the command but smaller.” It is a typed, effect-free observation of the current composition state.

Preview semantics by stage:

For a base selection, preview returns top rows, count if cheap, count exactness, retrieval lane, index readiness, and query description.

For field/value completion, preview returns candidate values with counts and source freshness, not rows.

For `group by`, preview returns group buckets, bucket counts, whether counts are exact, and omitted bucket count.

For `count`, preview returns count and exactness. If exact count is expensive, return bounded lower bound or approximate with clear label.

For `sort`, preview returns top rows after sort and the sort key distribution if useful.

For `limit/offset`, preview returns the visible page and total/exactness.

For unit-source queries (`messages where ...`, `actions where ...`), preview must show grain prominently. A message-row preview and a session-row preview are not interchangeable.

For set operations, preview is where the feature becomes understandable: show left count, right count, intersection count, left-only/right-only samples, and result count. This should be the default set-op affordance.

For projection, preview renders a bounded sample of the projected output and lists what was included, folded, redacted, omitted, or unavailable.

For render, preview shows the rendered form under the selected layout but capped.

For actions, preview is dry-run only: “would tag 12 sessions,” “would delete 0 because safety guard missing,” “would export 50 rows to file.”

The envelope should include `relation_grain`, `identity_key`, `preview_rows`, `exact_count`, `lower_bound_count`, `omitted_reason`, `degraded_reasons`, `freshness_epoch`, `cache_key`, and `elapsed_ms`.

Preview must degrade loudly. “No results” and “index not ready” are different states.

## Report C13 — incremental/debounced execution

The composer needs an event loop discipline, or it will feel worse than a normal CLI.

Recommended timing:

Parse on every edit, budget under 5–10ms.

Completion request after a tiny debounce, 25–75ms.

Preview request after a larger debounce, 100–200ms, and only when the parse state is valid enough to produce a relation.

Cancel stale preview requests immediately. The daemon should support cancellation by request id. If cancellation is not yet implemented, the client should still ignore stale responses using monotonically increasing request ids.

Use “latest wins.” Never let an old slow preview overwrite a newer one.

Cache layers:

Text → parse AST.

AST/spec fingerprint → relation preview.

Field/prefix/scope → value candidates.

Projection spec + sample ids → rendered preview.

Archive generation/convergence epoch invalidates caches. The daemon should include epochs in every response. If the archive changes while the user is composing, preview can say “stale; refresh pending” rather than flicker.

Execution budgets:

Completions: <20ms for grammar/static completions, <50ms for value completions from cached facets.

Preview: first paint under 100ms when possible, otherwise show count/plan first and rows later.

Long-running previews should stream partial events: parse OK, executing, first rows, counts, final. This is where a JSON-RPC/LSP-like protocol or SSE-style event stream becomes valuable.

Avoid exact counts on every keystroke. Exact count is a requested preview mode, not the default. Default should be bounded sample + approximate/lower-bound count + exactness marker.

## Report C14 — compose projection and render too

The composer should not stop at row selection. The real power comes from interactively shaping evidence and output.

Today named views are too central. Long-term, built-in views should become presets over a general projection/render algebra. `transcript`, `dialogue`, `context-image`, `timeline`, etc. should be default config rows or registry presets, not the only way to express projection.

Projection primitives:

Families: sessions, messages, blocks, actions, raw, context, chronicle, neighbors, correlation, temporal, assertions, variants, attachments.

Fields: arbitrary projectable payload fields, including parent/session fields when unit rows are selected.

Body policy: full, authored dialogue, omit tool outputs, metadata only, plus future custom policies.

Block filters: include/exclude block kinds, roles, tool families, code/diff/thinking/tool output.

Windows: neighbor windows, time windows, message windows around hits, lineage prefix/tail windows.

Budget: max tokens, max body bytes, body offset, edge limit, sample limit.

Folding: summarize/fold long tool outputs, collapse code blocks, fold repeated system/developer context, fold attachments.

Variant policy: original only, preferred language, show aligned variants, show both with alignment.

Evidence policy: include provenance, completeness flags, source refs, stale/degraded warnings.

Render primitives:

Encoding: markdown, JSON, NDJSON, HTML, org, yaml, plaintext, csv.

Layout: table, cards, transcript, dialogue, timeline, diff, cockpit, compact, verbose.

Destination: terminal, stdout, browser, clipboard, file.

The composer should let the user flip these with immediate preview: “show messages only,” “omit tool outputs,” “include actions,” “render as timeline,” “save as view.” The saved result is a named view/config macro.

This directly supports your “less named views, more generality” direction.

## Report D15 — grammar-driven completion

The current completion system is registry-based, which is good, but it is not yet cursor-aware structural completion. It asks for a completion `kind`; the client has to know whether it wants fields, structural units, projection fields, pipeline stages, etc. That is not enough for a powerful composer.

The desired system accepts `(input_text, cursor_offset, context)` and returns candidates. The daemon, not the client, determines the completion context.

Completion contexts:

Root command / query start.

Compact query field name.

Field value.

Boolean operator.

Inside `exists unit(...)`.

Inside `seq(...)`.

After terminal source: `messages where`, `actions where`.

Pipeline stage position after `|`.

Inside `with unit(...)` projection clause.

Inside set-op operand.

Inside macro reference `@`.

After `then` action connector.

Projection/render mode if the composer is editing a structured spec rather than text.

The engine should be grammar-derived but pragmatic. Use the Lark grammar where possible for parse-state expectations; use hand-written context detectors where the grammar is intentionally split outside Lark (`|` pipeline splitter, trailing `with`, render/projection syntax). The invariant is not “zero heuristics.” The invariant is “one authoritative completion service; no shell/client-specific vocabularies.”

Every candidate should carry: insertion text, replacement span, display, kind, group, description, source registry, score, danger flag, stale flag, optional count/sample, optional preview command.

Completion should not silently include unsupported features. If set algebra is designed but not executable, candidates can appear only if marked unsupported/experimental, or not at all.

## Report D16 — value-completion sources

Dynamic value completion is where the composer starts feeling magical. Static grammar completion says “origin:”; value completion says “origin:claude-code-session — 18,234 sessions.”

High-value fields:

Origins / source families.

Models and providers.

Repos and project paths.

CWD prefixes.

Tags.

Tools / actions / action sequences.

Message/block kinds.

Languages / variant preferences.

Session phases / work-event kinds.

Public refs / recently viewed refs.

Saved macros, views, cohorts.

Import source names and configured source roots.

Cost/usage dimensions: billing lane, provider, model family.

These completions must be index-backed or cached. Do not scan the archive per keystroke. The right substrate is a daemon-maintained value index/facet cache keyed by field, prefix, optional scope spec, archive epoch, and freshness. The current facets infrastructure can seed this, but composer completion wants lower latency and cursor-specific replacement spans.

API shape: `value.complete(field, prefix, scope_spec, limit, include_counts=True)`. Response includes candidates, counts, exactness, stale/degraded flags, and source.

Scoped value completion is important. If the query already says `repo:polylogue`, then model/tool/tag completions should reflect that scope when cheap. If scoped counts are expensive, return global candidates with a “global” marker rather than pretending they are scoped.

## Report E18 — projection algebra

Projection should become a first-class algebra, not a list of named views.

Core model: selection yields a relation. Projection maps relation + archive evidence into a shaped evidence bundle. Render maps that bundle into an output artifact.

Projection operations:

`fields`: choose columns/payload fields.

`attach`: attach related evidence families/units.

`window`: include surrounding context around hits.

`body`: choose body policy.

`filter-blocks`: include/exclude block kinds, roles, tools.

`fold`: collapse long/repetitive/low-signal parts.

`budget`: enforce token/byte/row/edge budgets.

`redact`: paths/secrets/private data.

`variants`: choose original/transformed/aligned variants.

`provenance`: include refs/completeness/staleness.

`evidence-tier`: raw/derived/asserted/variant/summary.

Syntax can be pipeline-like:

`find repo:polylogue | project messages(text, role, time), actions(tool_name) | body authored-dialogue | fold tool-output | budget 4000 tokens | render markdown layout:dialogue`

But the syntax is secondary. The key is the typed internal model. Named views become aliases for projection programs.

Composition rules matter. Some projection transforms commute; others do not. `redact` should be late and non-removable. `fold` before `budget` differs from `budget` before `fold`. The algebra should preserve order where order matters and normalize only safe commutative parts.

This is also where construct validity belongs: a projection that includes a derived insight should be able to include its evidence/provenance tier. Projection should not make derived data look like raw evidence.

## Report E19 — config/macro-defined views and layouts

There should be a single configurable preset system, not separate mechanisms for query macros, read views, render layouts, and saved cohorts.

Preset kinds:

Predicate macro: expands to a query predicate.

Relation macro: expands to a whole relation pipeline.

Projection preset: expands to a projection program.

Render layout preset: expands to a render spec/layout.

View preset: expands to a full `(Selection, Projection, Render)` triple or partial triple.

Cohort: frozen or re-evaluated result set.

Action preset: probably future, and must be safety-guarded.

Storage belongs in `user.db`, because these are human/operator assertions and preferences. Built-ins can be seeded as read-only presets or code-backed defaults, but the runtime should treat them through the same registry interface as user presets.

Expansion rules:

Validate at definition time.

Store normalized AST/spec plus original text.

Track provenance so `explain` can show `@coding` expanded to `origin:claude-code-session repo:polylogue ...`.

Prevent cycles with depth limits.

Version presets by schema/protocol version.

Allow export/import.

Allow override/disable of built-ins only explicitly.

This would directly satisfy your “less named views while enabling generality” direction. A view stops being a hardcoded command branch and becomes a named preset over the algebra.

## Report E20 — complete relation-algebra vision

The long-term read model is relation algebra over archive evidence.

A relation has: grain, identity key, columns, rank/order, evidence provenance, freshness, and completeness.

Grains form a lattice: sessions/logical sessions, messages, blocks, actions, files, runs, assertions, attachments, variants, work-events, threads. You can lower from sessions to messages/actions/blocks, and lift from messages/actions back to sessions. Lift/lower must be explicit when ambiguity matters.

Operators:

Selection: `where`.

Boolean predicates: and/or/not inside selection.

Existential predicates: `exists`.

Sequence: ordered occurrence.

Set ops: union/intersect/except.

Projection: fields/evidence families/body/render shaping.

Attach/join: controlled joins along known archive relationships, not arbitrary SQL join.

Group/aggregate: count, sum, avg, percentiles, min/max, distinct.

Having: filter groups by aggregate.

Sort/limit/offset.

Window: surrounding rows, time windows, lineage windows.

Lift/lower: change grain.

Sample: bounded representative rows.

Explain: show plan, grain, indexes, degraded state.

This does not mean exposing SQL. It means every powerful read affordance has a named algebraic operator with typed semantics. If a future feature cannot be expressed as this algebra, either the algebra is missing a primitive or the feature is not a read-surface primitive.

The danger is over-general joins. Polylogue should prefer relationship-aware attach/lift/lower over arbitrary cross products. The archive has a graph; the DSL should expose that graph safely.

## Report E21 — set-algebra at message/unit grain

Message-grain set algebra is harder than session-grain and must be designed explicitly.

Identity choices:

Session grain: probably logical session id or public session ref depending on lineage mode. Physical session id is sometimes wrong for dedup/lineage truth.

Message grain: default should be evidence identity, not text content. Use canonical public message ref / canonical row identity. Two messages with identical text are not the same message.

Lineage-aware message identity: a prefix-shared message may appear across physical sessions but represent the same logical content. There should be a mode or context where identity is canonical content/logical message ref, not physical row.

Content identity: optional, explicit `by content` semantics. Do not make it default.

Unit grains: actions/files/runs/assertions each need stable identity keys. If no stable identity exists, set ops fail until one is defined.

Ranking:

`union`: combine ranks using RRF or a rank-fusion policy; keep operand membership metadata.

`intersect`: preserve left operand rank by default; annotate right-rank. This makes `A | intersect (B)` mean “A, filtered by B.”

`except`: preserve left rank.

`union all` should probably not exist initially; dedup is the safer default.

Grain mismatch:

Fail unless there is an explicit lift/lower. `messages where text:auth | intersect (sessions where repo:polylogue)` should error. The user must write something like `messages where text:auth | sessions | intersect (sessions where repo:polylogue)` or use a scoped query form.

Preview for set ops should show left-only, both, right-only. That makes the algebra understandable.

## Report E22 — saved cohort primitive

Saved cohorts are the durable companion to set algebra.

Two distinct things should not be conflated:

Macro cohort: saved query/pipeline that re-evaluates against current archive.

Snapshot cohort: frozen row identities captured at a point in archive time.

Both are useful. Macro cohorts are “my Claude Code sessions last 30d.” Snapshot cohorts are “the 12 uplift-arm sessions used in finding X.” External findings and reproducibility require snapshots, not re-evaluated macros.

Cohort record should store: name, kind, grain, identity keys, source query/spec, archive identity/epoch, created_at, created_by, description, row count, optional metrics, optional frozen row list, stale/rebase state.

Cohorts compose with set algebra:

`@uplift_pack intersect @has_tool_errors`

`@arm_a except @arm_b`

`@finding_2026_07_v24 | read`

Preview should show cohort freshness: frozen cohort from archive epoch N, current archive epoch M, missing rows, new matching rows if re-evaluated.

Storage belongs in `user.db`. A cohort is an assertion/curation artifact, not derived index data.

## Report F23 — Go/Rust thin-client feasibility

A Go/Rust client is feasible only after the daemon protocol exists and the CLI has been thinned. Before that, it would either shell out to Python, embed Python, or reimplement substrate logic. All three are bad.

The useful architecture is: Python daemon owns substrate; Go/Rust client speaks UDS protocol. No FFI. No Python import from the client. No SQL in the client.

Why Go/Rust helps:

Near-zero startup relative to Python+Click.

Single static-ish binary distribution.

Better fit for always-snappy terminal UX.

Cleaner separation of client from substrate.

Why it may not be first:

The protocol is not stable yet.

A Python Textual/prompt_toolkit prototype can validate UX faster.

Polylogue already has Python UI contracts and Textual TUI code.

A second language adds packaging and contributor complexity.

Recommendation: prototype composer in Python first, but design the protocol as if a Go/Rust client will exist. Once the protocol and UX are stable, choose language based on desired UX shape. Go + Bubble Tea is a strong candidate for a fast stateful terminal app; Rust + Ratatui is a strong candidate for a lean, typed terminal app. citeturn319747search1turn319747search2

Do not choose “Go vs Rust” yet. Choose “wire protocol over UDS, codegen-able schemas, no substrate in client” now.

## Report G25 — prior art query builders and interactive search

The lesson from prior art is: the best query builders do not hide the query language. They make it inspectable, completable, previewable, and teachable.

Patterns to steal:

From LSP: diagnostics at cursor, completions from semantic context, capability negotiation, cancellation, and partial results.

From telescope/fuzzy finders: fast narrowing, visible candidate scoring, keyboard-first flow.

From Splunk/Kibana-style systems: show counts/facets as the user scopes; make time/range filters visible; let users save searches.

From PromQL-like builders: textual expression remains authoritative; UI can assist without replacing text.

From Datasette-like exploration: facets and value counts are discovery tools, not just filters.

From jq-like pipelines: every transform should have a clear input/output shape.

Anti-patterns:

Form-only builders that cannot express the full language.

Autocomplete that guesses strings without grammar state.

Preview that silently switches grain or hides stale indexes.

Saved searches that are opaque strings with no provenance.

Recommendation for Polylogue: build a textual-first structural composer. The user writes/edits the DSL, but the UI supplies grammar-aware completions, value counts, previews, explain, and save-as macro/view/cohort. This fits Polylogue’s operator/developer audience better than a form builder.

## Report G30 — what the daemon/protocol unlocks

The daemon protocol unlocks more than a faster CLI.

Immediate unlocks:

Live query composer.

Daemon-served shell completion.

Thin CLI startup.

Warm preview.

Value completion.

Protocol-level explain.

Query/action dry-runs.

Web workbench live updates.

Mission-control surfaces over agent coordination.

Medium-term unlocks:

Editor integration: `.polylogue` query files with completions/diagnostics, essentially an LSP-like mode for the DSL.

MCP context packs generated from saved macros/cohorts/projections.

Notifications/watch mode: “tell me when new sessions match this query.”

Saved cohorts as external finding artifacts.

Background projection materialization.

Local agent context scheduler using the same query/projection algebra.

Cross-surface parity: CLI, TUI, MCP, web, and API all ask the daemon for the same parsed/executed relation.

Long-term unlocks:

A real workbench where search/analyze/audit/remember are not separate commands but different render/action paths over one relation algebra.

A powerful “system of record” UX: the user composes evidence, previews it, saves it, cites it, and feeds it to agents without switching conceptual systems.

The warm daemon is therefore not a performance optimization. It is the runtime for the whole composable product.

## Cross-report synthesis

The build should be sequenced like this.

First, `t46`: thin the CLI and delete parallel query/read engines. This is not optional. Without it, every future client inherits divergence. The CLI must become a presentation/client layer over the same contract used by daemon/API/TUI/MCP.

Second, define the local daemon protocol. Start with UDS HTTP if easiest, but model it as versioned RPC envelopes. Include hello/capabilities, archive identity, query parse, completion, preview, execute, action affordances, status, and cancellation.

Third, make the daemon lifecycle product-grade: ensure/autostart, per-archive sockets, pid/lock cleanup, explicit degraded states, archive mismatch safety.

Fourth, build structural completion: cursor-aware parse-state completion, using existing registries and extending current completion candidates rather than replacing them.

Fifth, add archive-backed value completion with caches/facets and strict latency budgets.

Sixth, build the composer vertical slice: selection + completion + preview for session-grain queries, group/count, read. Use the daemon protocol. Keep the textual DSL visible.

Seventh, generalize projection/render: projection programs, config-defined views/layouts, named presets as data, not branches.

Eighth, land set algebra and saved cohorts. These make the composer truly algebraic and make findings reproducible.

Ninth, consider a Go/Rust client after the protocol is stable. Until then, a Python TUI is the right learning vehicle.

The demo should be framed as **“Orchestrator Rhetoric and Delegation Analysis”**, with the Fable tweet as the hook. It should not be a Fable-specific feature. Fable is merely one corpus slice. The generic object is: an orchestrator delegates work to subordinate agents or model families, using prompts, constraints, routing rules, output contracts, reprimands, retry instructions, and synthesis instructions. Polylogue should be able to extract those communications, build a delegation graph, annotate them, compare cohorts, and render a citable report.

The demo claim could be:

“Given a local archive of coding-agent sessions, Polylogue can find every subagent delegation, reconstruct the parent/subagent/report edges, classify the orchestrator’s instruction style, compare it by model family and task type, and produce an evidence-backed report with quoted specimens, aggregate counts, and links back to the original session evidence.”

The important constraint: the demo must only claim what the local logs show. It should not infer Fable’s private internal prompt or Anthropic’s hidden orchestration policy. The object of study is the visible harness output: task prompts, subagent system/developer instructions if captured, task descriptions, report files, scratchpads, terminal/tool actions, and final synthesis. That makes the finding honest.

A good report title would be something like:

or, less meme-like:

“Delegation Style in Multi-Agent Coding Sessions: A Polylogue Analysis”

The report would have two layers. First, the “hilarious examples” layer: short quoted specimens of strict delegation language, grouped by rhetorical pattern. Second, the serious analytic layer: counts and comparisons by orchestrator model, subagent model family, task type, outcome, and retry/failure behavior.

A nice specimen taxonomy:

“Directive density”: how many imperatives, constraints, and explicit prohibitions appear per delegation.

“Scope policing”: instructions like “do not edit,” “read-only,” “return JSON only,” “do not implement,” “do not summarize,” “ground every claim.”

“Output-contract strictness”: schema requirements, file-path requirements, evidence citations, line-number demands, exact artifact names.

“Trust posture”: whether the orchestrator assumes the subagent might drift, hallucinate, overreach, mutate state, or ignore hierarchy.

“Model-family accommodation”: whether prompts differ when sending work to Claude-family, OpenAI-family, Gemini-family, or cheaper/smaller subagents.

“Escalation behavior”: how the orchestrator reacts when subagents return weak output, fail, time out, or contradict each other.

“Delegation economy”: number of subagents spawned, token/cost estimates if available, and how much synthesis overhead follows.

The biggest missing piece is a **delegation graph**. Today Polylogue has sessions, messages, actions, blocks, assertions, raw/context/chronicle families, lineage, and work-event concepts. For this demo, we need a generic unit:

`delegation`

A delegation is not merely a message. It is a relationship: parent agent asks subordinate agent/tool/model to do a bounded task, with a prompt, scope, model identity, input context, output artifact, completion status, and subsequent parent use.

Schema-level shape:

```text
delegation_id
parent_session_ref
parent_message_ref / action_ref
subagent_session_ref nullable
subagent_model
subagent_model_family
task_title
task_description
instruction_payload_ref
output_artifact_ref
result_message_ref
status
started_at
finished_at
cost_estimate
provenance
extractor_version
confidence
```

This is generic enough to cover Claude Code subagents, Codex parallel tasks, Gemini agents, MCP-launched agents, future Polylogue-managed swarms, and even human-delegated tasks. It should not be named after Fable.

The second missing piece is **artifact linking**. The previous swarm itself produced scratchpad files, markdown reports, findings JSON, ledgers, and prompts. A delegation analysis often needs to connect:

parent session message → background agent launch → scratchpad report → final synthesis → Bead mutation / PR / commit

If Polylogue only sees the transcript but not the scratchpad/report file as a linked artifact, the evidence is fragmented. For this demo, scratchpad markdowns are first-class evidence. They need stable refs, content hashes, session links, and artifact family classifications.

The third missing piece is **external annotations as first-class evidence-adjacent assertions**. The sentiment/tone/directive classifications should not be stored as raw truth in `index.db`. They are derived judgments produced by an analyzer with a schema, model, prompt, evidence refs, confidence, and timestamp. They belong in `user.db` or an assertion/judgment table.

This is the difference between a toy demo and a construct-valid analysis. “Fable is authoritarian” is not a raw fact. “This analyzer labeled delegation X as directive_intensity=3 because of these evidence spans” is a judgment. Polylogue should preserve that distinction. That aligns with your existing construct-validity discipline: derived claims should carry their measurement method and evidence, not pretend to be raw observation. fileciteturn2file0

The fourth missing piece is **relation algebra over units**, not just session search. This demo wants queries like:

```text
delegations where orchestrator_model:Fable
  | group by subagent_model_family
  | count
```

```text
delegations where control_device:read-only
  | intersect (delegations where subagent_model_family:openai)
  | project instruction,result,parent-use
  | render specimen-gallery
```

```text
messages where role:orchestrator and directive_intensity:>=3
  | lift delegations
  | group by task_type
```

```text
delegations where status:failed
  | sequence delegate -> retry -> critique
```

This is where the earlier set-algebra/projection/cohort design becomes directly useful. The demo wants saved cohorts like `@fable_delegations`, `@non_claude_subagents`, `@strict_prompts`, `@failed_delegations`. It wants set difference: “Fable delegations except Claude-family subagents.” It wants intersection: “strict delegations that were later contradicted.” It wants lift/lower between delegation, message, action, artifact, and session grain.

The fifth missing piece is **projection algebra**. Named views will not be enough. For this demo, the evidence unit is a “delegation card”:

```text

delegation prompt

annotations
provenance refs
```

```text

  parent_use.window(after:3),
```

Built-in views can then become config-defined presets over this projection algebra. `transcript`, `dialogue`, `delegation-card`, `specimen-gallery`, `comparison-table`, and `evidence-pack` should be macros/layouts, not permanently special command branches. That is exactly the “less named views, more generality” direction.

How many subagents does Fable spawn per high-level request?

Which model families get which task types?

How many tokens/cost units does delegation add?

Does Fable use stricter prompts when sending work to cheaper/weaker models?

Do stricter prompts correlate with better completion or less cleanup?

This ties directly into provider-usage and cost-honesty work. It would make the demo more than anecdotal.

A label schema with definitions.

Calibration examples.

Evidence spans required for every label.

Uncertainty/abstain allowed.

Random sample audit.

Counterexample search.

Inter-rater comparison if two models classify the same cohort.

## First: what is a cohort?

A **cohort** is a named set of units that you can reuse as an operand in later analysis.

For Polylogue, the crucial detail is that a cohort has a **grain**. A cohort might be a set of sessions, messages, actions, tool calls, assertions, commits, artifacts, or query runs. A session cohort and a message cohort are not interchangeable.

There are two kinds:

A **dynamic cohort** is a saved query/spec. It re-evaluates against the current archive. Example: “all Fable-visible delegations to non-Claude subagents.”

A **snapshot cohort** is a frozen list of identities captured at a specific archive epoch. Example: “the 47 delegation events used in report v1.” This is what you cite in a published finding.

Existing `recall_pack` is nearby but not the same. A recall pack is an explicit bundle of items resolved into payloads; a cohort is a reusable relation/set with grain, provenance, freshness, and set-algebra behavior. In code, recall packs already resolve item refs like sessions/messages/annotations/marks and preserve degraded/missing counts, so they are a useful ancestor, but they are not yet a first-class query operand or relation object.

## Query objects are the next big unlock

Your “Polylogue tracks its own use” idea is very powerful, and I think it should be promoted from side thought to core substrate.

Right now, a query is mostly an invocation: parse, execute, render, disappear. The better model is:

`query object → query run → result relation → projection/render artifact → downstream annotation/cohort/report`

A **query object** is the durable or semi-durable representation of the question: raw text, parsed AST/spec, selection/projection/render specs, surface used, author, defaults, archive identity/epoch, and maybe parent query.

A **query run** is an execution event: when it ran, against which archive state, timing, degraded state, exactness, result grain, count, sample ids, result fingerprint, and output artifact refs.

A **query result relation** is the thing later analyses should be able to reference. It has grain and identity members. It can be dynamic, snapshot, sampled, capped, or exact.

This gives you queries like:

```text
from query:q_2026_07_05_001
| intersect @strict_delegations
| project delegation-card
```

or:

```text
queries where author_kind:agent and parent_analysis:a_fable_rhetoric
| group by status,degraded_reason
```

or:

```text
assertions where scope:query:q_123 and kind:judgment
```

The crucial policy distinction: not every CLI invocation belongs in durable `user.db`. Routine telemetry can go to `ops.db`; saved/promoted query objects, cohorts, analysis runs, and citation-bearing report inputs belong in `user.db`. Otherwise the archive drowns in its own usage traces.

The A2 protocol report already points in this direction: the warm daemon should own resident `Polylogue`, read pools, value-completion indexes, and shared handlers for HTTP/UDS, with `query`, `complete`, `preview`, and `status` latency budgets. That makes query-run recording cheap and consistent across CLI/MCP/TUI/web. fileciteturn10file8 The B8 report’s contract proposal is the right hard boundary: a thin client should speak verbs like `query`, `read`, `preview`, `complete`, `act`, `status`, and probably `facets`, not reimplement substrate logic. fileciteturn9file1

## “Analysis itself” should be represented as a provenance DAG

I would not model analysis as just YAML, and I would not model it as just database rows.

Use both:

A **recipe** is versionable, portable, and promptable. It can live as YAML/Markdown/Python/prompt file. It says: here is the intended analysis protocol.

An **analysis run** is database state. It says: this recipe/prompt was run by this actor/model/tooling at this time, issuing these queries, producing these cohorts, importing these annotation batches, generating these artifacts, and ending in this report.

A useful `analysis_run` object would have:

`analysis_id`, `recipe_ref`, `title`, `question`, `operator_ref`, `agent_ref`, `status`, `started_at`, `finished_at`, `archive_epoch`, `input_refs`, `query_run_refs`, `cohort_refs`, `annotation_batch_refs`, `artifact_refs`, `report_refs`, `parent_analysis_id`, `error/degraded_state`, and `summary`.

Then `analysis_step` objects form the DAG:

`select corpus → export evidence pack → annotate → import assertions → aggregate → outlier query → render report`.

This avoids trying to create a universal YAML workflow language up front. Complex analyses can be loops: an agent looks at results, decides what to query next, imports annotations, branches, retries, etc. The database records what actually happened. The recipe/prompt records what was intended.

This also answers the “queries-as-objects enables?” intuition: yes. Query objects are the nodes that make an analysis DAG useful rather than a prose blob.

For the Fable-style demo, the external agent should not just write `annotations.jsonl` and then a report. It should import an **annotation batch** into Polylogue.

A batch has:

`batch_id`, `schema_id`, `analysis_id`, `author_kind=agent`, `author_ref`, `model/provider`, `prompt_ref`, `created_at`, `target_grain`, `input_cohort_ref`, `row_count`, `accepted_count`, `rejected_count`, and validation/degraded info.

```json
{
```

The current assertion schema already supports almost all of that. The main gaps are ergonomic and contractual:

There should be a schema registry for annotation shapes.

That is very close to the desired analysis loop.

One naming point: “user annotations” is now too narrow. These are **overlay assertions** or **judgments**, written by humans or agents. The `author_kind` field already makes that distinction possible.

## Generic artifact linking: not scratchpad, not report-specific

The right unit is something like:

`artifact_observation`

with fields:

`artifact_ref`, `uri/path/blob_ref`, `repo_ref`, `commit_ref`, `content_hash`, `kind`, `mime/type`, `producer_ref`, `consumer_ref`, `session_ref`, `message_ref`, `action_ref`, `analysis_ref`, `created/observed_at`, `evidence_refs`, and `confidence`.

The source tier already has raw artifact infrastructure: `raw_artifacts` includes `artifact_kind`, `support_status`, `cohort_id`, `link_group_key`, and `sidecar_agent_type` (`polylogue/storage/sqlite/archive_tiers/source.py:84-102`). It also has raw sessions, blob refs, hook events, OTLP spans, and history sidecars (`source.py:15-156`). That is the low-level substrate. What is missing is the high-level relation and UX: “show me artifacts produced by this action/session/analysis,” “link this report to the query/cohort it summarizes,” “project session with produced artifacts,” “use artifact content as evidence.”

So the general model is:

`session/message/action/block/run/query/analysis → produced|consumed|mentioned|derived_from → artifact`.

A `.agent/scratch/.../report.md` is then just one artifact instance. A Claude Code report, Codex patch file, JSON findings file, exported evidence pack, generated chart, markdown report, or PR description is the same kind of thing.

## Repo files, commits, and replay/reproduction

You remembered correctly: Polylogue is already close to supporting a work-reconstruction layer.

In the code snapshot, sessions already carry git-ish fields like `git_branch`, `git_repository_url`, and `commit_hash` in the `sessions` table (`polylogue/storage/sqlite/archive_tiers/index.py:39-75`). There are repo tables: `session_working_dirs`, `repos`, `session_repos`, and `session_commits`; `session_commits` includes `commit_sha`, `repo_id`, `detection_type`, `method`, `confidence`, and `evidence_json` (`index.py:470-520`). Actions are paired from tool use/result blocks and include tool command/path, error state, exit code, and semantic type (`index.py:324-343`; action row shape in `archive.py:322-340`). There are also `session_runs`, `session_observed_events`, and `session_context_snapshots` (`index.py:913-1024`).

That is enough for a first **work trace** projection:

`session → ordered actions/tool calls → files touched → git commits associated → outputs/errors → final claims`.

But it is not enough for true replay unless file/repo state is captured more deeply.

For replay/reproduction, Polylogue would need to correlate or ingest:

pre/post git status snapshots,

commit diffs,

working tree file snapshots or content hashes around tool calls,

command cwd/env,

tool inputs and outputs,

exit status and stderr,

branch/head/index state,

external artifacts generated,

possibly Sinex terminal/process/filesystem events for the machine-level truth.

Then Polylogue can produce:

`reconstruction`: “what happened”

`reproduction plan`: “how to attempt rerun”

`replay bundle`: “checkout commit X, apply artifact Y, run commands Z”

`claim-vs-evidence audit`: “agent said it passed, but command exit code was 1”

I would be careful with the word “replay.” Actual replay may be impossible or unsafe for many sessions. “Reconstruct” and “reproduce” are safer product concepts. Replay can be an optional dry-run/plan artifact, not the default promise.

This connects strongly to the coordination envelope. The current coordination code builds a bounded envelope from repo, self, work item, beads, process/resource, handoff, archive evidence, session trees/activity/proof/context refs (`polylogue/coordination/envelope.py:60-116`; payloads in `polylogue/coordination/payloads.py:14-219`). It already resolves sessions from env tokens like `POLYLOGUE_SESSION_REF`, `CODEX_SESSION_ID`, and `CLAUDE_SESSION_ID` and archive evidence (`coordination/envelope.py:655-734`). That is operationally useful, but it is not yet a durable work-trace graph. The next layer should persist the trace.

## Stored query/result objects would make agent loops much less fiddly

The agent workflow you described becomes clean if every query returns an id.

Instead of:

“run CLI, parse stdout, write notes somewhere, query again manually”

the agent can do:

1. `polylogue query "delegations where orchestrator:Fable" --json`
2. receive `query_run_ref`, `result_relation_ref`, `sample`, `count`, `grain`
3. `polylogue evidence-pack result:<id> --projection delegation-card`
4. annotate evidence pack externally
5. `polylogue assertions import annotations.jsonl --scope analysis:<id>`
6. `polylogue query "assertions where scope:analysis:<id> and key:directive_intensity and value>=4"`
7. save a cohort or render report

That can happen via CLI if the CLI is a daemon-fast thin client. MCP is not required. You are right to push back on “don’t shell out.” Shelling out is fine if `polylogue` is really a thin protocol client and returns stable refs/JSON. MCP is just another surface. The underlying affordance is not MCP; it is **stable object refs and typed envelopes**.

The B7 report is relevant because it says the CLI must stop owning substrate logic: extend read contracts, collapse `archive_query.py` to facade calls, add ops/write/diagnostics surfaces, relocate DTOs, and enforce no runtime imports from storage/pipeline/archive query internals. fileciteturn10file13 Without that, agents using the CLI will keep hitting weird divergence between CLI, daemon, MCP, and API.

## The missing high-level affordances

Here is the generic set I would add, in roughly the order they become useful.

First: **query runs**. Every execution has an id, parent id, result relation, grain, count, exactness, degraded state, and output artifact refs. Stored in `ops.db` by default, promotable to `user.db`.

Second: **result relations**. A result is a typed relation over unit refs, not just rows printed to stdout. It can be exact, sampled, capped, stale, or snapshot. It can be projected, saved, intersected, and cited.

Third: **cohorts**. Dynamic or snapshot named result sets, with grain and identity policy. They are the durable form of “this is the corpus I mean.”

Fourth: **annotation batches**. External agents import judgments as assertions under a schema and analysis scope.

Fifth: **analysis runs**. A higher-level DAG tying queries, cohorts, annotation batches, evidence packs, reports, and artifacts together.

Sixth: **evidence packs**. A projection artifact built for external analysis: bounded, cited, machine-readable, and optionally renderable for humans.

Seventh: **artifact graph**. Generic produced/consumed/mentioned/derived_from links between sessions/actions/queries/analyses and files/blobs/reports/commits.

Eighth: **work traces**. Reconstructed ordered traces of agent work across tool calls, filesystem/repo changes, commits, artifacts, and final claims.

Ninth: **report objects**. A report is not just Markdown. It is an artifact derived from an analysis run, with citations to evidence refs, cohorts, query runs, and annotation batches.

Tenth: **policy/promote layer**. Because not every query or trace should become durable human memory. The default can be ops telemetry; agents/humans can promote significant things into user durable state.

This is the “algebraic combinatorics over one-off rigid silos” lens from your raw log made concrete: instead of special commands for Fable, devloops, reports, translations, or scratchpads, Polylogue gets typed units and relations that compose. fileciteturn10file18

## Assertions vs Beads

Beads and assertions overlap but should not collapse into each other.

A **Bead** is a work item / directive / planning object. It has status, priority, deps, acceptance, design, and execution semantics.

An **assertion** is an overlay claim/judgment/annotation about some target. It can say “this session has pathology X,” “this message is a translation candidate,” “this query result supports finding Y,” “this decision is active,” or “this claim should be injected into context.”

Polylogue should be able to reference Beads and maybe ingest them as external work-item artifacts, but Beads remain the canonical task tracker. Assertions can attach to Beads:

`target_ref = bead:<id>`

`kind = judgment|blocker|lesson|decision|handoff`

That answers the raw-log question “how do assertions compare to beads?”: assertions are evidence/knowledge overlays; beads are executable work directives. They should link, not merge. fileciteturn10file12

The nice generalization is “devloops themselves could be expressed in Sinex/Polylogue/Beads.” A devloop is an analysis/work run: Beads define intended work, Polylogue records AI-session evidence, Sinex records machine activity, assertions record judgments/memory, and the analysis/work-trace object ties them together. fileciteturn10file17

## Content variants and “alternates” are not quite assertions

Your raw-log idea about alternate forms of messages, especially translations, is important: attach alternate content to message/block refs, allow rendering with a selected alternate, and clearly mark the substitution. fileciteturn10file15

I would not make these ordinary assertions long-term. They can be represented through assertions initially, but semantically they are **content variants**:

original message/block content remains raw evidence,

variant content has transform type, language, model/prompt/source, alignment refs, confidence, created_at,

render policy can choose original, translated, simplified, summarized, or side-by-side,

exports must disclose that variant substitution occurred.

This connects directly to the content-variants epic mentioned in the orientation/chatlog and the swarm brief. It also gives a non-Fable use case for the same external annotation/import machinery: external agents can produce variants, not just labels.

## Beads I would add or sharpen

I would not add a “Fable analysis” epic. I would add substrate beads that make that demo trivial.

**First-class query run recording.** Record every CLI/MCP/TUI/web query execution as a typed query-run envelope with query text, lowered spec, result grain, count/exactness, degraded state, timing, surface, author, parent query, and optional result sample. Return the `query_run_ref` in every JSON response.

**Result relation refs.** Introduce a durable/semi-durable object representing a query result relation over a grain, with exact/sampled/capped/snapshot status and member identity policy. Allow `from query:<id>` or `from result:<id>` in the DSL.

**Cohort primitive.** Dynamic and snapshot cohorts over any supported grain. Existing recall packs can remain human-curated bundles, but cohorts should be algebra operands.

**Annotation batch import.** Register schema, validate JSONL, import rows into assertions/judgments with evidence refs, batch refs, author/model/prompt metadata, and candidate/review status.

**Analysis run DAG.** Persist analysis runs and steps connecting query runs, cohorts, annotation batches, evidence packs, reports, and artifacts.

**Generic artifact graph.** Promote raw artifacts into queryable/linkable artifact refs, with produced/consumed/mentioned/derived_from edges from sessions/messages/actions/runs/queries/analyses.

**Repo file / commit / work trace projection.** Build a projection that reconstructs session work from actions, tool outcomes, files touched, repo state, associated commits, and generated artifacts. Do not promise deterministic replay yet; promise reconstruct/reproduce plan.

**Target kind expansion.** Add user-state target kinds for `query`, `query_result`, `analysis`, `analysis_step`, `annotation_batch`, `artifact`, `repo_file`, `commit`, maybe `bead` or generic `external_work_item`.

**Evidence-pack renderer.** Render bounded, machine-readable evidence packs from arbitrary result relations/projections, with stable refs and completeness/degraded flags.

**Agent affordance surface.** CLI/MCP verbs for `analysis start/step/finish`, `assertions import`, `cohort save`, `evidence pack`, `query explain`, and `report render`, all through the same daemon-fast contract.

**Ops-vs-durable policy.** Decide which self-observation goes to `ops.db`, which gets promoted to `user.db`, and how agents request promotion. This prevents query-run tracking from becoming noise.

## The core product sentence

Polylogue should become the substrate that turns messy agent traces into reusable typed analysis objects:

`raw sessions + repo/artifact evidence → query runs → result relations → cohorts → annotation batches/assertions → analysis runs → reports`

That is the generalization behind the Fable demo, the claim-vs-evidence demo, agent recovery, devloop analysis, and future methodological evals. The current code already contains many of the parts. The missing piece is to make the relations and provenance objects first-class, returned by every surface, and pleasant for agents to use.

The thing to notice in the Claude Code excerpt is that `/compact` created two different “memory layers.” First, Claude produced a normal compaction summary of the just-finished work: commits, branch handling, verification, and leftover branch cleanup. Then the pre-compact hook injected `bd prime --memories-only`, which dumped 19 persistent `bd remember` entries into the new context. Those entries include doctrine, traps, schema rules, CLI/daemon pitfalls, token semantics, and Beads workflow rules. The hook even warns that the output may be truncated and that the full persisted output may need to be read before continuing. fileciteturn17file15

That is already valuable. It is also exactly the kind of thing Polylogue should beat.

Claude’s current behavior is basically: “here is a flat text preamble of memories; please carry them.” Polylogue’s better behavior should be: “here is a task-scoped, evidence-linked, freshness-aware context object compiled from queryable project state; every included item has a reason, a target, provenance, and expansion affordances.”

## How far we are from beating this

Close for a minimal version. Medium distance for a genuinely good version. Farther for the best version, but the far version is not speculative in architecture; it is the natural endpoint of the bead-set.

The minimal version is surprisingly near because Polylogue already has the important primitive: `user.db.assertions`. The current schema/doc model treats `user.db` as the irreplaceable tier for user and agent overlays, and specifically says assertions cover marks, annotations, corrections, suppressions, tags, metadata, saved views, recall packs, workspaces, candidate claims, and judgments. Candidate lifecycle states such as candidate/accepted/rejected/deferred/superseded are already part of the model. fileciteturn17file3 The code snapshot also has assertion kinds like `saved_query`, `recall_pack`, `workspace_note`, `decision`, `caveat`, `lesson`, `blocker`, `handoff`, `judgment`, `run_state`, `prompt_eval`, `transform_candidate`, and `pathology`. So the storage concept is already much richer than `bd remember`.

What is not yet there is the good path from “these memories exist” to “an agent receives the right memory, in the right form, at the right time, with refs and freshness.” Today, Claude sees a flat list. Polylogue should expose a compiled context pack.

A very small first move would already beat Claude Code’s memory dump: import `bd memories` as Polylogue assertions, with `target_ref` pointing at the project/repo/workspace, `scope_ref` pointing at the bead/program/domain, `kind=lesson|caveat|decision|blocker|run_state`, and `context_policy.inject=true/false`. Then `polylogue context compile` could select relevant entries by current repo, current bead, branch, recent failures, and command surface. That would already be better because the agent would get a selected pack, not every memory every time.

The current bead-set already implies more than that. It frames Polylogue as “the system of record for AI work,” with search/analyze/audit/remember as the product verbs; it also identifies the direction as agent coordination, content variants, lineage truth, attachment integrity, scale-hardening, external legibility, a context scheduler, a web evidence cockpit, and outcome-grounded analytics with construct-validity metadata. fileciteturn17file3 That means the target is not “a nicer CLAUDE.md.” It is an operational memory substrate.

## What Claude’s flat memory gets wrong

The `bd prime` block is global. It does not know whether the next task is schema work, daemon work, query DSL, docs, cost accounting, or Beads hygiene. So every new context inherits the same mixture: automagic invariants, shell traps, schema bumps, devloop daemon gotchas, notes-sidecar trap, token semantics, etc. Some of those are always useful; many are conditional. A task working on query grammar should see the Lark/pipeline-stage trap. A task working on provider cost should see token semantics. A task editing Beads should see shell backtick substitution and notes-sidecar trap. A task touching README probably does not need SQLite inspection advice.

It is not evidence-linked. The memory says “token semantics: Codex input includes cached tokens,” but the agent cannot click through to the exact session, cost probe, query result, code path, or bead that established that. The memory says “notes sidecar trap,” but it is prose, not a queryable invariant over Beads. It says “schema bumps,” but not as a schema-change guard that can run.

It is not freshness-aware. Some entries are stable doctrine. Some are branch-local operational state. Some are pitfalls that should eventually be fixed and disappear. Some are “current live archive” warnings. Claude sees them all as plain text.

It is not auditable. After compaction, you cannot ask: which memories influenced this answer? Which were omitted because of budget? Which were stale? Which came from `bd remember`, from CLAUDE.md, from Beads, from docs, from observed failures, from a previous query, or from a human assertion?

It is not hierarchical. The hook says “if truncated, read the full persisted output,” but the memory itself does not carry structured expansion handles. A better memory card would carry a concise text body plus `expand_ref`s: source assertion, source bead, source query run, source file path, source session/message/action.

It is not a live view. The `beads-workflow` memory says Beads is the durable backlog/directive substrate and GitHub issues are external refs, not mirrors. That is true and important. But Polylogue can do better by querying the actual bead graph, current branch, recent commits, and open dependencies, then compiling the relevant subset. The bead-set characterization explicitly says Beads is program-shaped, not a flat list, with epic towers and terminal states; that structure is exactly what should drive context selection. fileciteturn17file16

## What “better” should look like

A Polylogue-compiled post-compaction context should look less like a memo and more like a small evidence pack.

For example, after `/compact`, instead of injecting 19 memories verbatim, Polylogue would produce something like:

“Current task: branch cleanup / docs readiness. Active work object: PR #2546/#2547. Current repo state: master clean. Relevant durable doctrine: Beads are authoritative over GitHub; execution-grade beads; notes-sidecar trap. Relevant operational hazards: .claude/settings archive-root pitfall only if querying live archive; zsh backtick substitution only if editing bead text. Recently verified facts: devtools verify exit 0; branch force-with-lease preserved readiness commit. Excluded: token semantics, schema bump policy, SQLite inspection, because current task is not cost/storage/query debugging. Expansion refs: …”

That is a qualitative jump. The agent gets less text, more usable state, and the ability to expand.

The strongest version would also show “why included” and “what to do next.” Not just “execution-grade beads,” but:

“Because the user asked to delete branches and read session 6a49…, include: live-archive-root pitfall; daemon PID/API info; SQLite inspection caveat; read-completely doctrine; scratch/artifact query refs.”

That is exactly the kind of selection Claude’s flat hook cannot perform.

## Against the current bead-set scope

The current full bead-set already points to most of this, but it is distributed across several epics rather than unified under a “compaction/context memory” product concept.

The direct ingredients are:

Agent coordination substrate: a typed coordination envelope over sessions, topology, repos, work items, and resources, degrading from Beads to git/GitHub/session inference. That is the “what is this agent currently doing?” layer. fileciteturn17file3

Context/memory loop: the bead-set explicitly has a context scheduler as the single arbiter of what enters an agent’s context. fileciteturn17file3

Assertions/judgments: user.db already owns durable human/agent overlays. fileciteturn17file3

Query DSL and read algebra: the search surface already supports `assertions where ...` and other unit queries; docs list assertion fields such as author, author_kind, evidence, key, kind, scope, scope_ref, status, target, target_ref, text, value, and visibility. That means memories can be queried as first-class units, not just injected as prose. fileciteturn17file3

Hot daemon + thin client: the current design says the daemon should be the required warm path for speed/composer, while direct substrate remains only as break-glass/library/test mode. This is important because context compilation should be a daemon operation, not a slow shell-script assembly. fileciteturn17file11

Composer/history/recall: the C10 design already says every committed composer run writes a recall entry to `user.db` with query text, resolved spec, result fingerprint, and timestamp; named recall entries become macros, and recalled queries store resolved specs rather than only text. fileciteturn17file1 That is close to your “query as object” idea, but it needs to be generalized.

So under current scope, we are not missing the philosophical substrate. We are missing the connective product layer.

I would define the missing current-scope feature as:

**Context compiler over assertions, Beads, repo state, query history, and session lineage.**

It should compile the exact thing Claude Code is trying to fake with `bd prime`, but do it from typed relations.

## Where the current designs are still too weak

The biggest weakness is that “recall entry” is too small a concept. C10 says every commit writes a recall entry to `user.db`. That is right as a user-facing history/macro feature, but too weak as a substrate. A recall entry is not enough to represent analysis or compaction continuity.

I would split it into four objects:

`query_definition`: reusable dynamic question/spec. This is the thing that can become a macro.

`query_run`: a specific execution against archive state, with actor, time, surface, parsed spec, archive epoch, degraded state, timing, count/exactness, sample ids, and parent task/analysis/session.

`result_relation`: the typed set returned by the query: grain, identity policy, exact/sampled/capped/snapshot state, member ids or fingerprint, and freshness.

`render_artifact`: the human/machine output produced from a result relation: markdown, JSON, evidence pack, report, clipboard content, browser view, etc.

Only the first one belongs naturally in durable `user.db` by default. The raw run stream belongs in `ops.db` by default and becomes durable only when promoted, cited, used in an analysis, saved as a cohort, or attached to a report. Otherwise Polylogue will drown in its own command history.

Second weakness: assertions need batch/schema ergonomics. The storage can already hold judgments, prompt evals, lessons, caveats, etc., but an agent needs something like `assertions import --schema directive_style_v1 annotations.jsonl`, with validation, evidence-ref checks, candidate/accepted lifecycle, and a batch object. Without batch/schema support, external annotations are technically possible but fiddly.

Third weakness: context policy is underdeveloped. The table has `context_policy_json`, but the product needs an actual policy language:

inject when `repo=polylogue`

inject when target bead is descendant of `fnm`

inject only after failed command containing `bd create`

inject only if freshness query passes

inject as warning, not doctrine

include summary only; expand on demand

never inject automatically; available by search only

This is where the Claude memory block becomes obsolete. A memory should not merely exist; it should know when it wants to be in context.

Fourth weakness: compaction itself is not yet a first-class event. Polylogue has lineage/compaction work in scope, but the product layer should explicitly model compaction boundaries:

`compaction_event`: source session, compacted summary, injected memories, omitted refs, continuation session, model/harness, timestamp, and post-compaction first actions.

Then you can query:

“Which compacted sessions later failed because a memory was missing?”

“Which persistent memories are always injected but never used?”

“Which post-compaction sessions immediately ask for repo orientation again?”

“Which memories are stale or contradicted by current Beads/docs?”

That is how you beat Claude’s `/compact`: not by summarizing better, but by making compaction auditable.

## The best design

The best design is not “Polylogue writes a better CLAUDE.md.” It is:

**Polylogue treats context as a compiled projection from a typed memory/work graph.**

The graph has these main node types:

sessions, messages, blocks, actions, tool results, compactions, agents, runs, Beads/work-items, PRs, commits, files/artifacts, queries, query runs, result relations, assertions/judgments, cohorts, context snapshots, analysis runs, reports.

The edges are:

produced, consumed, mentioned, derived_from, supports, contradicts, supersedes, scoped_to, depends_on, blocked_by, included_in_context, omitted_from_context, verified_by, failed_by, continued_by.

Then a context preamble is not a document. It is a render of a `context_snapshot` object.

The render can be Claude-facing prose, MCP JSON, CLI markdown, or a web cockpit. But internally it is a selection/projection/render over the memory graph.

This gives you far more power than Claude Code’s bead memories:

You can compile per-task memory. A schema migration agent gets schema doctrine and active migrations. A query DSL agent gets Lark/pipeline traps. A cost analytics agent gets token semantics and pricing-source truth. A Beads-editing agent gets notes-sidecar and zsh quoting traps.

You can compile per-risk memory. If the next action is `bd create --design`, include the shell backtick substitution trap. If the next action queries the live archive, include `POLYLOGUE_ARCHIVE_ROOT` pitfall. If the next action touches cost, include Codex cached-token semantics.

You can compile per-state memory. If branch-local devloop daemon is running from stale code, include restart warning. If prod daemon is inactive, include deploy-trust limitation. If the archive is mid-rebuild, include derived-tier freshness warning.

You can compile evidence-grounded memory. Every item has a source assertion, source query run, source bead, source session, or source file. The agent can ask to expand instead of relying on prose.

You can compile contradiction-aware memory. If one memory says durable tiers never migrate in place and another says schema-evolution-v2 added additive migrations, the compiler can refuse to inject both as doctrine and instead surface a reconciliation warning. That exact docs/schema drift pattern appears in the uploaded bead audit. fileciteturn17file19

You can compile budgeted memory. Rather than “19 memories dumped,” it selects a token budget across doctrine, task state, hazards, recent work, and expansion refs.

You can measure memory utility. Because context snapshots are recorded, Polylogue can later ask: which injected memories were followed, ignored, contradicted, or led to successful work?

## The specific design I would add

I would add a **Context Compile subsystem** with three durable-ish object families.

First, `memory_assertion`, implemented using existing assertions. Do not invent a silo. Use `kind=lesson|caveat|decision|blocker|handoff|run_state|judgment`, with key namespaces such as:

`doctrine.automagic_invariants`

`pitfall.shell_backtick_substitution`

`pitfall.live_archive_root`

`schema.schema_evolution_v2`

`cost.codex_cached_tokens`

`workflow.execution_grade_beads`

Add schema validation and context policy on top, not a new “memory” table.

Second, `context_compile_run`, likely in `ops.db` by default. It records:

actor/session, task/work-item, requested budget, query/specs used, candidate memory ids, selected memory ids, omitted memory ids, omission reasons, rendered artifact hash, archive/beads/git state fingerprints, degraded/freshness warnings.

A promoted context compile can become durable if used in a report, handoff, or published finding.

Third, `context_pack` or `memory_pack`. This is a reusable dynamic selection policy, not a static blob. Examples:

`@beads_editor_pack`

`@schema_change_pack`

`@cost_analysis_pack`

`@query_dsl_pack`

`@live_archive_operator_pack`

Each pack is a query/projection over assertions + Beads + repo state + recent sessions. Claude’s current `bd prime --memories-only` is just `@global_project_pack render claude-preamble`, but the better default is task-specific.

Then add `compaction_boundary` as a run/session unit. This can be derived from Claude Code logs when `/compact` appears. It should store the compact summary, injected memory pack, source session, continuation segment, and evidence refs. It becomes queryable:

```text
compactions where repo:polylogue AND injected:key:token-semantics
```

```text
sessions where after_compaction:true
  | project context-compile, first-actions, failures
```

```text
assertions where context_policy.inject:true AND stale:true
```

The interface should be boring and direct.

Shows why each item was included.

Searches assertions/memories with evidence and freshness.

`polylogue memory revalidate --all --changed-since master`

Runs refresh checks for memories that carry verifier queries.

Shows the compaction boundary, summary, injected memories, and continuation.

## How much better it gets

It gets better along five dimensions.

First, relevance. Claude’s hook includes 19 memories. Polylogue should include the 5–8 relevant ones and leave the rest as expandable refs.

Second, correctness. A memory like “schema-evolution-v2 uses additive migrations for durable tiers” should be verified against docs/schema/code or marked stale. Claude just trusts the string.

Third, actionability. “Notes sidecar trap” should not merely warn the agent. It should cause `bd show --json | .notes` to be part of the Beads-editing workflow, or trigger a lint if the agent fills acceptance while notes contains content.

Fourth, provenance. Every memory should answer “where did this come from?” and “what would change my mind?” This aligns with Polylogue’s construct-validity posture.

Fifth, self-improvement. Because Polylogue tracks context compile runs and downstream outcomes, it can discover that some memories are noise, some are missing, and some are too late. Claude’s flat memory cannot learn this except through manual editing.

## How this ties to query-as-object

Your query-as-object idea is central. The C10 design already gestures at it with recall entries, macros, result fingerprints, and resolved specs. fileciteturn17file1 But the design should be upgraded from “recall entries for composer history” to “query runs are substrate events.”

A query result should be a relation object. A context pack can then include “the result of query Q,” not a pasted list. A memory can be scoped to a query. An assertion can target a query result. A report can cite a query run. A later agent can say:

```text
from query_run:q_20260705_1234
| project evidence-card
```

or:

```text
assertions where scope:context_compile:c_123 and status:candidate
```

or:

```text
query_runs where actor_kind:agent AND degraded:true
| group by degraded_reason
```

This is how “analysis itself” becomes representable without forcing every analysis into YAML. Recipes/prompts describe intended procedure; query runs, assertions, cohorts, context compiles, and artifacts record what actually happened.

## What is missing from the bead-set

I would add or sharpen these beads.

**Context compile runs.** Record every context compilation as an object: input task/session, candidate memories, selected memories, omitted memories, reasons, token budget, rendered artifact, freshness/degraded state.

**Memory assertions import from Beads.** Import `bd memories` / `bd remember` output into `user.db.assertions`, preserving keys, scope, evidence, author, timestamps, and context policy. The first version can be one-way import; later versions can reconcile.

**Context policy schema.** Define `context_policy_json` semantics: inject conditions, priority, budget class, expiry, verifier query, audience, expansion behavior.

**Compaction boundary unit.** Extract `/compact` events from Claude Code sessions, including compact summary, injected memory pack, source/continuation session refs, and post-compaction first actions.

**Context pack renderer.** Render a selected context pack for Claude Code, Codex, MCP, Markdown, and JSON, with refs and omitted-item counts.

**Context explain/diff.** Given a context snapshot, show why each item was included; compare two context snapshots across compaction or handoff.

**Query run/result relation objects.** Promote C10 recall entries into a split query_definition/query_run/result_relation/render_artifact model, with ops-vs-user retention.

**Assertion batch import.** External agents can annotate messages/sessions/query results and import judgments under a schema; batch metadata records prompt/model/evidence/validation.

**Memory revalidation lane.** For each active injectable memory with a verifier, run checks and mark stale/conflicted/superseded. Examples: schema doctrine vs docs, Beads decisions still open, branch daemon stale, archive root env.

**Agent context success analytics.** Measure whether injected memories correlate with fewer repeated mistakes, faster recovery after compaction, less “orientation” time, fewer wrong archive queries, fewer bead-editing traps.

## The answer in one sentence

Claude Code currently survives compaction by dumping a flat list of persistent memories into context; Polylogue is close to beating that at the storage/query level, medium distance from beating it as a product, and should ultimately replace it with a context compiler over typed assertions, Beads, repo state, query runs, compaction boundaries, and evidence-linked memory packs.

The right bar for these demos is: **a stranger can inspect the artifact and see a task get easier, truer, or more recoverable because Polylogue exists.** The situation brief is explicit that the current demo shelf mostly proves the substrate is honest, while the missing thing is an agent or operator completing a real task better; it names browser-captured agent recovery and claim-versus-evidence as the closest high-leverage demos. fileciteturn22file4

Below are many grounded demo ideas. I’m deliberately biasing toward things that use real existing traces: Claude Code compaction, Beads, claim/evidence data, browser capture, raw logs, agent swarms, commits, sessions, tool calls, cost data, lineage, and context packs.

## Highest-value demos to ship first

1. **Cut-session recovery pack**

A coding agent is handed a session id and a terse goal: “continue this work.” Baseline: agent gets only the user’s manual summary. Polylogue version: agent gets a recovery pack with prior session summary, relevant messages, tool calls, files touched, commits, open beads, branch state, and unresolved risks. Output: the next agent successfully performs a bounded follow-up without the user re-explaining. This directly matches the situation brief’s closest-to-done demo: browser-captured agent recovery. fileciteturn22file4

2. **Claim-versus-evidence report**

Run a corpus analysis over coding-agent sessions: find places where the assistant claimed success, but tool calls, exit codes, CI logs, failed tests, or verifier output contradicted it. Output: a report with counts by model/provider/tool, a specimen gallery, and exact evidence refs. This is probably the most externally legible “finding travels on its own” demo: “coding agents silently proceed past failed calls N% of the time.” fileciteturn21file11

3. **Claude compaction memory audit**

Use the attached Claude Code excerpt as the hook. After compaction, Claude receives a flat dump of 19 persistent Beads memories, including global doctrine, shell traps, schema rules, daemon gotchas, and cost semantics. Polylogue demo: compile a task-specific context pack from the same memory/assertion material, explain why each memory was included or omitted, and compare token load plus downstream usefulness. fileciteturn22file2

4. **“Fable as Foreman” subagent-rhetoric report**

The tweet hook becomes a general analysis: extract subagent delegations, classify instruction style, output-contract strictness, scope policing, retry behavior, and model-family routing. Output: funny specimen gallery plus serious aggregate tables. This uses the same external-annotation loop discussed earlier: an agent labels delegation events, imports judgments/assertions, then queries them.

5. **Beads swarm autopsy**

Take the huge Beads audit swarm as data. Reconstruct: prompts issued, domains assigned, subagent outputs, contradictions between auditors, critic gate decisions, applied mutations, held items, and final ledger. Output: “How a 40-agent backlog audit actually worked.” This is a process-story demo, and the situation brief says the process story is the safest and most ready inbound story. fileciteturn22file19

6. **Before/after Beads quality audit**

Input: pre-audit Beads snapshot. Polylogue extracts null acceptance criteria, orphaned beads, missing dependency edges, stale GitHub refs, and under-specified frontier work. Output: before/after dashboard: acceptance criteria filled, new beads created, deps added, cycles avoided, held decisions. This turns the Beads restructuring into a concrete “agentic project management substrate” demo.

7. **Stale docs versus code reality check**

Given a stale architecture doc and live code/schema, Polylogue finds contradictions and produces a PR-ready patch plan. The uploaded swarm/chat excerpt contains exactly this pattern: architecture-spine schema-versioning contradicted additive durable migrations and execution-plan was stale. Output: stale-doc report with code/doc/bead refs and recommended patches. fileciteturn22file15

8. **Critical-path selector**

Input: the whole Beads graph. Ask: “What should I work on next?” Polylogue produces a ranked frontier: ready, blocked, stale, high-leverage, proof-campaign relevant, and anti-recommendations. Output: a short decision memo, not a giant list. This is grounded in the audit’s conclusion that the real wave-1 collapsed to a small frontier like `cfk`, `3tl.11`, `3tl.4`, `1xc.*`, `pj8`, and a few 4smp entries. fileciteturn21file15

9. **Notes-sidecar trap detector**

Demo a specific Beads failure mode: many beads looked under-specified because important content lived in the notes sidecar rather than design/acceptance fields. Polylogue finds these, lifts/formalizes them, and prevents agents from inventing duplicate specs. Output: “hidden spec recovery” report plus patch commands.

10. **GitHub external-ref reconciliation**

Beads are authoritative; GitHub issues are external refs, not mirrors. Demo: Polylogue compares Beads and GitHub issue state, identifies done issues still open on GitHub, stale refs, issue scopes folded into larger beads, and non-mirror relationships. Output: operator checklist, no automatic GH mutation. This is grounded in the Beads workflow memory in the compaction excerpt. fileciteturn22file2

## Demos centered on the live query/composer vision

11. **Live set-algebra explainer**

A TUI or recorded terminal demo: compose `A intersect B`, see `|A|`, `|B|`, and `|A∩B|` update before commit, then export the result. The C10 design explicitly frames live set sizes as the payoff of set-algebra preview because users otherwise guess wrong and discover empty results too late. fileciteturn22file0

12. **Projection preview: selection is not enough**

Show the same query rendered as `summary`, `dialogue`, `tool-failures-only`, `commit-card`, and `evidence-pack`. The demo claim: Polylogue is not just finding sessions; it is composing evidence. C10’s commit model already says terminal stages preview their projection, and export reruns the same render spec so preview and artifact match. fileciteturn22file0

13. **“Find touches:ids.py” composer walkthrough**

Use C10’s concrete example. User types `find touches:ids.py |`, stage rail shows the upstream cardinality, completion suggests `group by`, `sort`, `limit`, `intersect`, `union`, `except`, `read`, and preview shows the current relation before the next stage is chosen. Output: a short screen recording showing query literacy emerging from the UI. fileciteturn22file11

14. **Query recall becomes macro**

Run a query, commit it, then reverse-search it with `Ctrl-R`, preview it, name it, and use it as `@macro` in another set expression. This grounds your “query objects” idea in the existing C10 design, which says committed runs write recall entries with query text, resolved spec, result fingerprint, and timestamp, and named recall entries become macros. fileciteturn22file1

15. **Value completion over the real archive**

Type `origin:cod...`, `model:cla...`, `tool:rg...`, `tag:...`, and show values plus counts from the live archive. C10 calls this the honest gap: current completions cover structure, but field-value completion needs live archive-backed values. fileciteturn21file6

16. **Safe destructive action**

A deliberately theatrical demo: query a set, type `| delete` or `| mark`, and Polylogue refuses to do anything until it shows affected-row count and asks for confirmation. This demonstrates the algebra/action model and safety posture at once. C10 explicitly says preview never executes destructive terminals and commit gates them on preview-backed confirmation. fileciteturn22file0

17. **Explain-at-point-of-use**

User types an ugly query. Polylogue shows parse tree, grain, retrieval lane, pipeline stages, projection, exactness/capped count, and why the current preview is stale or sampled. Output: query debugging as a teaching surface. This is a better demo than generic “autocomplete,” because it shows the archive being inspectable.

18. **Daemon-fast CLI versus cold CLI**

Record the same query run cold and then through the daemon-backed thin client. But do not make it a benchmark-only demo. Make the claim: because the warm daemon makes preview/completion possible, the CLI becomes an interactive instrument. A3 says the composer/live-preview UX only exists honestly with the warm daemon, while daemonless remains break-glass/substrate-direct. fileciteturn22file5

## Demos centered on annotations/assertions

19. **External annotation loop**

Prompt an agent: “Label these 200 delegation prompts for directive intensity, output-contract strictness, and scope control. Evidence refs required.” Import JSONL as assertion/judgment candidates. Then query the annotations. Output: report generated from external labels, not manual spreadsheet fiddling.

20. **Assertion provenance demo**

Take one judgment, such as “this assistant ignored a failing test,” and expand it: target message, evidence refs, tool result, confidence, author/model, schema, batch, status. This demonstrates why assertions are better than CLAUDE.md or flat memories. The raw log explicitly notes that assertion entries could be timestamped, carry expiry, have navigable origin refs, versioning, and hierarchical expansion refs. fileciteturn22file8

21. **Human-vs-agent annotation disagreement**

Two models label the same cohort of messages for “claim unsupported,” “scope drift,” or “overconfident closeout.” Polylogue shows agreement, disagreements, and specimen rows. Output: construct-validity artifact, not just labels.

22. **Pathology atlas**

Create an annotation schema for agent failure modes: ignored error, false completion, excessive scope, repeated dead-end, under-specified plan, stale doc reliance, unsafe shell quoting, wrong archive root. Run it over a sample and produce an atlas with examples and counts.

23. **Correction lifecycle**

An agent imports candidate assertions. Human accepts/rejects a few. Polylogue then changes what gets injected into future context packs. Output: one loop from observation → candidate memory → accepted assertion → context behavior.

24. **Finding package**

Select a cohort, import annotations, generate a report, and include every dependency: query run refs, snapshot cohort, annotation batch, evidence pack, render artifact. This is the “analysis itself as object” demo.

## What these demos imply Polylogue should build

The demos mostly converge on the same missing primitives. Not Fable-specific, not scratchpad-specific.

Polylogue needs **query runs** as first-class objects: every query returns an id, lowered spec, result grain, exactness, degraded state, timing, and result fingerprint. C10 already proposes recall entries with query text, resolved spec, result fingerprint, and timestamp; that should grow into query-run/result-relation objects. fileciteturn22file1

It needs **cohorts**: named dynamic or snapshot sets of sessions/messages/actions/delegations/assertions/artifacts. These make demos reproducible: “the 47 examples in report v1” is a snapshot cohort, not a vague search.

It needs **annotation batches** over assertions: external agents label messages/sessions/delegations/query results, import JSONL, and Polylogue makes those judgments queryable with schema, author, confidence, evidence refs, and status. The raw-log assertion idea already points to timestamped, expiring, ref-backed, versioned memory/assertion entries. fileciteturn22file8

It needs **artifact/work-trace linking**: sessions, actions, files, commits, PRs, reports, evidence packs, and generated artifacts linked by produced/consumed/derived_from edges. This is what makes replay plans, commit archaeology, and agent-swarm autopsies work.

It needs **analysis runs**: not just YAML recipes, not just database rows. A recipe/prompt describes intended procedure; the analysis run records what actually happened: query runs, cohorts, annotation batches, evidence packs, reports, and decisions.

It needs **context compile runs**: every context pack has selected memories, omitted memories, reasons, refs, freshness, and downstream outcome. This directly beats flat `bd prime` memory dumps after compaction.

## My ranking

The first public demo should be either **claim-versus-evidence** or **cut-session recovery**. The first is the best public finding. The second is the best agent-value proof. The third, slightly more playful but still serious, is **Fable/subagent-rhetoric**.

The best internal dogfood demo is **compaction memory replacement**, because the Claude excerpt gives you a perfect before/after: flat persistent memories versus a compiled, cited, task-scoped context pack. The best substrate-shaping demo is **external annotation loop**, because once that works, almost every other analysis becomes scalable instead of fiddly.
