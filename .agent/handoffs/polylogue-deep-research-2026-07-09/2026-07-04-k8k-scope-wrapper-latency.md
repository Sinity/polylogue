# k8k Scope Wrapper Latency

## Question

Does the Polylogue devshell wrapper add durable, large latency to lightweight
CLI reads by routing `polylogue` through `sinnix-scope background`?

## Wrapper Resolution

```text
polylogue is /realm/project/polylogue/.direnv/sinnix-scope/bin/polylogue
polylogue is /realm/project/polylogue/.venv/bin/polylogue
polylogue is /etc/profiles/per-user/sinity/bin/polylogue
sinnix-scope is /etc/profiles/per-user/sinity/bin/sinnix-scope
```

The wrapper is generated by `/realm/project/sinnix/scripts/sinnix-direnvrc`.
For Polylogue it wraps `pytest`, `uv`, `nix`, `make`, `sqlite3`, `duckdb`,
and `polylogue`. The wrapper classifies `polylogue`, `sqlite3`, and `duckdb`
as `background`; other unlisted commands default to `build`.

## Host State

`sinnix-observe` at the start of the run showed no blocked tasks and
`polylogued.service` inactive. Host pressure was not pristine: memory PSI was
around 0.4 to 0.8 and I/O PSI around 4 to 18 over the sampled windows, with
about 8.3 GiB memory available. That is enough to record as background pressure,
but not enough to explain a reproducible 45s wrapper tax in the fresh timing
matrix.

## Classification

No durable wrapper-policy bug reproduced. The current overhead from
`sinnix-scope` is tens of milliseconds for a trivial command and roughly
indistinguishable from `uv run`/direct invocation on the real usage-headline
read. The original 45.59s observation is best classified as transient host or
systemd/scope pressure at the time of the earlier run, not a current Polylogue
or Sinnix wrapper design failure.

No Sinnix code change is justified by this evidence. If the 45s symptom
returns, the next probe should capture the exact `sinnix-scope` unit name and
journal while the slow command is still active, because the steady-state wrapper
policy is not currently the bottleneck.
