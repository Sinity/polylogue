# Decision map

## Sinex

### Recall-first — recommended repository front door

Core promise: **Ask your machine what you were doing.**

Use for the general README and readers who need practical payoff before accepting the infrastructure cost. The root `sinexctl recall` command is already shipped and is the shortest bridge from “event substrate” to a problem a stranger recognizes.

Main dependency: a public artifact owner, not a new recall engine. Keep media watermarked until a scrubbed packet or deterministic analog lands.

### Rebuildable-first — strongest concept/launch narrative

Core promise: **Your data should not be interpreted only once.**

Use for a concepts page, launch essay, conference/demo page, or technical audience already interested in personal data infrastructure. It names the deepest difference from capture tools: source evidence stays stable while interpretations are versioned computations that can be rebuilt, compared, archived, and judged.

It is not the default action because readers may retain the analogy without knowing what to do. Keep a concrete recall or replay receipt nearby.

### Replay-first — best narrow technical differentiation

Core promise: **Capture once. Rebuild understanding later.**

Use for architecture pages and event-sourcing/data-infrastructure readers. Append-only evidence plus explicit reinterpretation is more distinctive than generic activity capture, but a new reader must first understand why machine history matters.

### Event-spine-first — conservative contributor positioning

Core promise: **The event spine for a local machine.**

Use when the repository is primarily recruiting infrastructure contributors. It is weaker as a general front door because it asks the reader to care about implementation before showing a recovered decision, failure, or work window.

### Sinex combinations to avoid

Do not place recall, replay, privacy, outage diagnosis, semantic retrieval, and agent resumption in one first screen. They are a proof shelf, not one promise.

Do not treat the older direct-SQL recall demo as the current product surface. Use `sinexctl recall`; retain older packets only as bounded method evidence.

Do not publish private scale observations as universal benchmarks. Do not show a precise gap interval when the typed contract only carries an unbounded caveat. Do not remove the prototype watermark until media comes from a released fixture through the typed owner.

## Polylogue

### Receipts-first — recommended repository front door

Core promise: **Know what the agents actually did.**

Use for the README, social preview, release notes, and public demo page. “Evidence system” becomes concrete in one glance, and the anti-grep control answers the strongest skeptic.

### Search-first — fastest first action

Core promise: **Search every AI-work history as one archive.**

Use when discovery traffic arrives from archive-search intent. Risk: readers may collapse the project into “cross-provider chat search,” underselling structural actions, evidence refs, lineage, and reviewed context.

### Continuity-first — agent-memory/handoff audience

Core promise: **Give the next agent evidence, not a summary.**

Use for MCP/agent integration material. Capability can be described now; measured downstream improvement cannot be implied before the paired experiment reports.

### Polylogue combinations to avoid

Do not put all providers, five user questions, six badges, a large screenshot, local-first caveats, and multiple install routes before the first proof.

Do not lead with “flight recorder” without qualification. Do not hand-edit visual evidence. Media must be generated from real routes and private-data-free fixtures, with a drift gate connecting recipe, normalized output, and committed artifact.

## Shared selection rule

Choose one hero based on the question the first proof can answer now. Promote a composition only when its claim, generated evidence, clean-machine action, explicit boundary, and cold-reader result agree; a structural score alone never decides.
